I've been implementing user story 26 with claude agent teams. I have some findings that should lead to improvements.

What happened:

- Used Opus 4.6 to create a plan
- Started a new Sonnet session to implement the plan "Implement the following plan. Use an agent team to parallelize the work, keeping the context small and commit in small parts. Every commit should verify with npm run verify:quiet. `@.context/attachments/plan.md`
- The main session started work, assigned some agent teams and built the feature

Observations:

- Sonnet created a list of tasks that could happen in parallel. It assigned 4 agents, but after that it kept idling most agents and re-using only one agent to do most of the work. After asking why, it realized the mistake and started to spawn more agents.
- The plan that was created included creating a lot of tests first. This led to a red state of the app, but all agents understood this and worked with it. I did not like it, I want every commit to be a complete unit that leaves the app in a working state.
- Since mulitple agents were working at the same worktree at the same time and verified their own work, there was a lot of time wasted on seeing changes they did not make, fixing commits etc. I don't yet know if I want the overhead of multiple branches, but this does not work with a bigger feature like this one.
- Agents were running npm run verify:quiet as instructed, which gave them too little insights into what failed ("Probably lint, I'll run that to see"). Rerunning things that just ran is a time and token waster. Also the verify was often broken due to other working parallel agents
- Since agents were parallelizing, the next build often failed on a lockfile since another agent was also building.
- The main thread used very little context, it stayed under 40% the entire time. This is great. However I don't have good insights into the rest - we'll have to analyze the run.
- The tasks were batched by technology (first creating all tests, then creating all api routes etc.). This led to two problems: 1. There are commits that leave the app in a non-working state. 2. Certain patterns were not good in the code, but were repeated and needed to be reworked. We should have checked in earlier to see if the code generated was qualitative. The plan was not clear enough on the implementation. The main issue is that the db is used directly in api routes, which should have been in services. It's fine that we figure this out and we can fix stuff in claude.md etc for the future, but it's a waste we found this after many work was done.
- It's assigning task to idle agents. I wonder if this is good practice when looking at token usage: we want clean, focused agents with clean contexts instead of piling on tasks. ("[SONNET] We have 10 failing component tests that need fetch mocking instead of localStorage. Let me create tasks for these and assign them to our idle agents")
- Maybe Sonnet is not very skilled at managing teams and we should try with Opus for the main thread. Since it's a relative low token user this might be a good investment: high effort on orchestrating, focused specialists on executing.
- Commits and verify runs are happening in a Bash() task, but the full output is added to the context of the team member agents (I suspect). These use general purpose agents with way too many tools etc. included. Creating specialists agents for certain jobs that happen a lot should help there. We need to analyze a bit with these hypotheseses and see where we can win big. I'm planning to create a few agents with specific tool access (on a need-to-know basis, so start with nothing and add when needed), that reuse skills. That means other agents can use the skills as well, but the specialists do it the most effective. We can also move some claude.md or other context to the specialists to focus them even more.
